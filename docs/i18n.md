# Internationalization (i18n) & locale persistence

This app uses **[next-intl](https://next-intl.dev)** for translations and URL-based locales. **localStorage** and a **`NEXT_LOCALE` cookie** keep the user’s language choice aligned with the route across visits and full page loads.

---

## Supported locales

| Code | Notes |
|------|--------|
| `en` | Default locale (`lib/i18n/config.ts` → `defaultLocale`) |
| `fr` | French |

Message catalogs live in:

- `lib/i18n/messages/en.json`
- `lib/i18n/messages/fr.json`

Namespaces are nested JSON keys (e.g. `marketing.nav.brand`, `admin.users.deleteConfirmTitle`).

---

## Routing & URLs

Configured in `lib/i18n/routing.ts`:

- **`localePrefix: 'as-needed'`** — English often has **no** prefix (`/`, `/about`), French uses a prefix (`/fr`, `/fr/about`).
- **`defaultLocale: 'en'`** — unprefixed paths are treated as English unless the middleware resolves otherwise.

**Middleware** (`middleware.ts`) runs `next-intl/middleware` so every matched request gets the correct locale and response headers. It also participates in **locale detection** (see [next-intl: Middleware](https://next-intl.dev/docs/routing/middleware)).

Navigation helpers (`Link`, `useRouter`, `usePathname`, …) come from `lib/i18n/navigation.ts` (`createNavigation(routing)`), not from `next/navigation` directly, so locale prefixes stay correct.

## How translations are loaded

### Server components & layouts

Use the server API so the active segment’s locale is explicit:

```ts
import { getTranslations } from 'next-intl/server';

const t = await getTranslations({ locale, namespace: 'marketing.landing' });
// or
const t = await getTranslations('admin.users');
```

`app/[locale]/layout.tsx` loads messages with **`getMessages({ locale })`** and passes them to **`NextIntlClientProvider`**. That pins the message bundle to the URL segment and avoids SSR/client mismatches.

### Client components

```tsx
import { useTranslations } from 'next-intl';

const t = useTranslations('admin.contact');
return <button>{t('deleteConfirmLabel')}</button>;
```

`useLocale()` returns the active locale from the provider (same as the current route).

### Request config

`lib/i18n/config.ts` exports **`getRequestConfig`** for the next-intl plugin. It loads `./messages/${locale}.json` based on `requestLocale`.

---

## Avoiding hydration mismatches (important)

1. **Always tie messages to the route locale** — e.g. `getMessages({ locale })` in `[locale]/layout.tsx`.
2. **Do not** read `localStorage` during render to pick a language; the **URL** is the source of truth for the first paint.
3. For marketing CTAs that were sensitive to SSR/client drift, copy is rendered via a **server component** (`StoreDownloadLinks`) and passed as **React nodes** into the client navbar instead of passing translated strings only as props into a client subtree.

## localStorage & cookie

### Keys and names

| Mechanism | Key / name | Purpose |
|-----------|------------|---------|
| **localStorage** | `pawtaker_locale` (`LOCALE_STORAGE_KEY`) | Persist choice in the browser for client-side logic |
| **Cookie** | `NEXT_LOCALE` (`LOCALE_COOKIE_NAME`) | Same name as **next-intl’s default** `localeCookie`, so **middleware** can read locale on **full page requests** before React runs |

Implementation:

- `lib/i18n/localeStorage.ts` — `getStoredLocale`, `setStoredLocale`
- `lib/i18n/localeCookie.ts` — `setLocaleCookie`, `getLocaleFromCookie`

**Whenever `setStoredLocale` runs**, it also calls **`setLocaleCookie`** so storage and cookie stay in sync.

## LocaleSync (`components/i18n/LocaleSync.tsx`)

Rendered once inside **`app/[locale]/layout.tsx`** (inside `NextIntlClientProvider`). It runs **only on the client** (`useEffect`).

Behavior:

1. **No value in localStorage**  
   - Try **`getLocaleFromCookie()`** — if `en`/`fr`, mirror into localStorage via `setStoredLocale` (also sets the cookie).  
   - Otherwise **`setStoredLocale(currentLocale)`** from the URL.

2. **Stored locale === route locale**  
   - Nothing else to do.

3. **Route is explicitly non-default** (e.g. `/fr/...`)  
   - **URL wins** — `setStoredLocale(currentLocale)` so bookmarks/shared links update the saved preference.

4. **Route is the default locale** (often unprefixed English) but **stored locale differs** (e.g. user previously chose French)  
   - **`router.replace(pathname, { locale: stored })`** after mount so the preferred language applies without breaking SSR (redirect happens after hydration).

---

## LocaleSelect (`components/i18n/LocaleSelect.tsx`)

Used in the marketing navbar and admin top bar. On change it:

1. Updates **`setStoredLocale(nextLocale)`** (localStorage + `NEXT_LOCALE` cookie).
2. Navigates with **`router.push(pathname, { locale: nextLocale })`** from `@/lib/i18n/navigation`.

A **`useEffect`** keeps the select value and storage aligned when **`useLocale()`** changes (e.g. after navigation).

## Summary flow

```text
User opens URL
    → Middleware + next-intl resolve locale (path, cookie, Accept-Language, …)
    → [locale]/layout loads messages for that locale
    → React renders with matching translations
    → LocaleSync (client) aligns localStorage + cookie with URL or redirects if needed
User changes language in LocaleSelect
    → localStorage + NEXT_LOCALE cookie updated
    → Client navigates to same path with new locale
```

## Sticky locale, `as-needed`, and `localeDetection`

For **unprefixed** URLs (e.g. `/` or `/about` for English), next-intl’s middleware resolves the locale in this order (see `next-intl`’s internal `resolveLocale`):

1. **Locale prefix in the path** (e.g. `/fr/...` → `fr`)
2. **`NEXT_LOCALE` cookie** — only if **`localeDetection` is not `false`**
3. **`Accept-Language`** — only if **`localeDetection` is not `false`**
4. **`defaultLocale`** (`en`)

So:

- **Keep `localeDetection: true` (the default)** — or omit it — so the middleware **reads `NEXT_LOCALE`**.  
  Setting **`localeDetection: false`** disables **both** the cookie **and** `Accept-Language` for detection; unprefixed paths then fall back to **`defaultLocale`**, which often looks like “refresh resets to English.” That setting is **not** the right fix for a missing sticky cookie in current next-intl.

This project also uses **`LocaleSync`** so that after hydration, if the URL is still the default locale but **localStorage** (and the cookie) say `fr`, the client navigates to `/fr/...` without breaking SSR on the first paint.

## Troubleshooting (refresh / navigation resets language)

1. **Cookies (Application → Cookies)**  
   After switching to French, confirm **`NEXT_LOCALE=fr`**: **Path** should be **`/`**, **SameSite** typically **Lax**, **not** HttpOnly (client code sets it via `document.cookie` in `setLocaleCookie`).

2. **Network → first document request** (`/` or `/about`)  
   Check **Request Headers** → **`Cookie`** includes `NEXT_LOCALE=fr`.  
   - If the cookie is **missing** here, the problem is **setting** the cookie (path, domain, or it was never written after switching).  
   - If the cookie **is** sent, middleware should resolve `fr` before React runs (unless the path already implies another locale).

3. **Code paths**  
   - **Writes:** `setStoredLocale` → `setLocaleCookie` (`lib/i18n/localeStorage.ts`, `lib/i18n/localeCookie.ts`).  
   - **Client redirect after load:** `LocaleSync` when the URL is default locale but storage says otherwise.  
   - **Do not** turn off **`localeDetection`** expecting the cookie to carry sticky locale — in next-intl it does the opposite for cookie-based detection.

## Adding a new string

1. Add the key under the right namespace in **`en.json`** and **`fr.json`** (keep structure parallel).
2. Use **`useTranslations('namespace')`** or **`getTranslations`** / **`getTranslations({ locale, namespace })`** on the server.
3. For new locales later: extend `locales` in `config.ts` and `routing.ts`, add `messages/<locale>.json`, and allow the value in `getStoredLocale` / cookie parsing if you validate strictly.
